# Development Guide

For developers and AI agents working on the Caros code. Pairs with [ARCHITECTURE.md](./ARCHITECTURE.md).

## Workflow

Caros is built in **phases** (`carosv2.md` is the execution plan). The convention:

> **one phase = one branch = one PR = reviewed before the next.** Don't start the next phase before the current is merged (build green + reviewed + committed). This keeps changes isolated.

- **Branch + tag per phase** (e.g. `chore/p2-schema-foundation`). Small, grouped commits; keep the build green at each commit.
- **Migrations are reviewed, then applied by hand** via the Supabase **SQL editor** (no `supabase db push` / Docker in this project). Every migration in `supabase/migrations/` is written **idempotent** (`IF [NOT] EXISTS`, drop-before-create on policies) and documents its apply order in a header comment.
- After a schema change, if PostgREST serves a stale schema cache, run:
  ```sql
  NOTIFY pgrst, 'reload schema';
  ```
- **Regenerate TS types** after schema changes (see below), then `npm run build`.
- **`npx tsx scripts/test-rls.ts` must be green** before merging anything touching schema, policies, or the leads/tenant access paths.

### Regenerating Supabase types (Windows — UTF-8 only)

`lib/supabase/database.types.ts` is generated by the Supabase CLI. On Windows PowerShell, **force UTF-8** — the default `>` redirect writes **UTF-16**, which breaks the TS build with confusing parse errors.

```powershell
# CORRECT — UTF-8
supabase gen types typescript --project-id <ref> --schema public |
  Out-File -Encoding utf8 lib/supabase/database.types.ts

# WRONG — default redirect writes UTF-16 LE (BOM) → build breaks
supabase gen types typescript ... > lib/supabase/database.types.ts
```

## Key conventions (do these)

- **Leads are write-only for the public.** When inserting a lead as anon, **never** chain `.select()` — anon has no SELECT policy on `leads`, so `INSERT … RETURNING` is denied (`42501`) and the row rolls back. See `lib/leads/submit.ts`. Resolve `tenant_id` from the `x-tenant-id` header server-side, never from client input.
- **UUIDs: use `safeRandomUUID()`** from `lib/utils/uuid.ts` — never call `crypto.randomUUID()` directly. `crypto.randomUUID` only exists in **secure contexts** (HTTPS/localhost); on a plain-HTTP origin like `dealer2.lvh.me` it's `undefined` and throws, crashing the page. The util falls back to `crypto.getRandomValues` (available on HTTP).
- **Per-tenant data flows through providers, not static config.** Read tenant data via `useTenantFeatures()` / the Tenant*Provider hooks (client) or `getTenantConfig()` / `getStorefrontFeatures()` (server). Don't reintroduce `config/*` or static `data/*` for tenant-varying values. (`data/cars.ts` was removed in P3 — the storefront is fully Supabase-backed via `lib/supabase/queries.server.ts`.)
- **Supabase clients (`lib/supabase/client.ts`):** `createPublicServerClient()` (anon, RLS) for storefront reads; `createBrowserClient()` (anon, **cookie** session via `@supabase/ssr`) for client auth; `createServerClient()` (**service role, bypasses RLS**) only for trusted server mutations / scripts — never for public reads.
- **Migrations:** idempotent, header-documented, applied via SQL editor. `ALTER TYPE … ADD VALUE` cannot run in a transaction that also *uses* the value — keep enum-extension migrations add-only (see `extend_market_enums`).
- **Definer functions** pin `search_path` (`set search_path = ''` + fully-qualified names) — see the harden migration and `mark_latest_lead_whatsapp`.

## Gotchas / hard-won lessons

| Symptom | Cause | Fix |
|---|---|---|
| TS build fails with garbled/parse errors in `database.types.ts` | type-gen written as **UTF-16** by `>` redirect | regenerate with `Out-File -Encoding utf8` |
| New column/enum "does not exist" via the API after a migration | PostgREST **schema cache** stale | `NOTIFY pgrst, 'reload schema';` |
| Page crashes with `crypto.randomUUID is not a function` on a `.lvh.me` (HTTP) origin | `randomUUID` missing on non-secure context | use `safeRandomUUID()` |
| Public lead insert silently fails / rolls back (`42501`) | `.select()`/RETURNING on an anon `leads` insert | write-only insert, no `.select()` |
| Sign-in appears to "hang" (dashboard bounces to login) | session in `localStorage`, invisible to the server | use `@supabase/ssr` cookie client (`createBrowserClient`) |
| Storefront 404 with a real subdomain | subdomain doesn't map to an active tenant (no silent default fallback) | check `tenants.slug`/`subdomain`/`active` |
| A "body type" shows `electric` | legacy `car_category='electric'` overlaps fuel | excluded in app code; full fix is the P3 remap |

```
app/
  [locale]/            storefront (ar/en): home, fleet, fleet/[id], booking, contact,
                       about, faq, services, financing, privacy, terms, LandingPage
    layout.tsx         tenant config load + brand CSS-var injection + providers + fonts
  dashboard/           authed dealer app: cars (CRUD), leads, settings, site
  auth/login/          Supabase auth
  sitemap.xml, robots.txt
components/
  hero/                storefront hero (HeroSection, HeroSearchPanel, results, chips, trust)
  home/                home sections (FeaturedSpotlight, WhyChooseUs, HowItWorks, FAQ, …)
  dashboard/           dashboard chrome + forms
  providers/           Tenant{Features,Contact,Pages,Content}Provider, Theme, UiLoading
  pages/, ui/          page clients + shared UI primitives
lib/
  supabase/            client.ts, getTenant.ts, queries.server.ts, mappers.ts, database.types.ts
  tenant/              features.ts, plans.ts, sections.ts, pages.ts, content.ts, branding.ts, resolveTenant.ts
  leads/               submit.ts (capture), schema.ts, markWhatsapp.ts
  search/              buildIndex, createSearch, searchVehicles, normalize (Fuse.js)
  vehicles/            options.ts (filter derivation), listingType.ts, filters/
  utils/               uuid.ts (safeRandomUUID), …
  fonts.ts             next/font definitions (Inter, IBM Plex Arabic, Space Grotesk, Cairo)
hooks/                 useHeroSearch, useSearchIndex, useDebouncedValue, …

messages/{en,ar}/      next-intl UI strings
config/                legacy/global non-tenant config (site, seo, features)
docs/                  these docs
```

## Before you merge — checklist
- [ ] `npm run build` green
- [ ] `npm run lint` clean
- [ ] `npx tsx scripts/test-rls.ts` → all passed (run the full proof with tenant-A creds)
- [ ] migrations idempotent + header-documented; applied to the DB; `NOTIFY pgrst` if needed
- [ ] types regenerated (UTF-8) if schema changed
- [ ] tested light/dark and a relevant tenant type (sale/rental/hybrid)
